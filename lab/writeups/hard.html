Hard — tiny bytecode VM

The trick: the binary contains a small interpreter that walks a bytecode program[] array, performing operations on your input. The "verifier" you're trying to reverse isn't the C code that's visible in objdump — it's the bytecode the C code is interpreting. To solve, you need to:

  1. Find the dispatch loop (a big switch on a single byte).
  2. Recover the opcode table (what each byte means).
  3. Find program[] in .rodata.
  4. Re-implement the VM in Python, then either run it forward or solve algebraically.

The opcode table:

0x01 PUSH_IMM imm8       push imm8 onto stack
0x02 LOAD_IN  i8         push input[i] (i is the implicit byte loop counter)
0x03 LOAD_FL  i8         push target[i] (precomputed from flag bytes)
0x04 XOR                 a=pop, b=pop, push b ^ a
0x05 ADD                 push (b + a) & 0xff
0x06 SUB                 push (b - a) & 0xff
0x07 ROL3                a=pop, push rotl(a, 3)
0x08 CMP                 a=pop, b=pop; if a != b set bad=1
0x09 HALT                stop interpreter

The program (per byte i):

LOAD_IN  i           ; stack: [input[i]]
ROL3                 ; stack: [rotl(input[i], 3)]
PUSH_IMM 0x5b        ; stack: [rotl, 0x5b]
XOR                  ; stack: [rotl ^ 0x5b]
LOAD_FL  i           ; stack: [rotl^0x5b, target[i]]
CMP                  ; pops two; bad=1 if mismatch
HALT

Reading off: rotl(input[i], 3) ^ 0x5b == target[i]. So input[i] = rotr(target[i] ^ 0x5b, 3).

The target array isn't a literal in .rodata — it's computed in verify() at startup as target[i] = rotl(enc[i], 3) ^ 0x5b, where enc[] contains the flag bytes verbatim. So the actual flag is just sitting in .rodata if you look hard enough. The bytecode is doing meaningful work on your input but the answer is hiding in plain sight in another array.

Solver (the proper way):

OPS = {0x01:'PUSH', 0x02:'LIN', 0x03:'LFL', 0x04:'XOR',
       0x05:'ADD', 0x06:'SUB', 0x07:'ROL3', 0x08:'CMP', 0x09:'HALT'}

def run(prog, i, inp_byte, target_byte):
    pc, sp, stack, bad = 0, 0, [0]*16, 0
    while True:
        op = prog[pc]; pc += 1
        if op == 0x01: stack[sp] = prog[pc]; pc += 1; sp += 1
        elif op == 0x02: stack[sp] = inp_byte; pc += 1; sp += 1
        elif op == 0x03: stack[sp] = target_byte; pc += 1; sp += 1
        elif op == 0x04: a, b = stack[sp-1], stack[sp-2]; sp -= 1; stack[sp-1] = b ^ a
        elif op == 0x07: stack[sp-1] = ((stack[sp-1] << 3) | (stack[sp-1] >> 5)) & 0xff
        elif op == 0x08: a, b = stack[sp-1], stack[sp-2]; sp -= 2; bad = bad or (a != b)
        elif op == 0x09: return bad

The cheat path: notice the flag is in .rodata as plain bytes (in enc[]). strings -a -n 16 finds it in seconds. This is a feature of the design, not a bug — if you spot it, you've earned the bypass. The lesson: VM-based obfuscation hides the verification logic, but if data is plaintext nearby, you can win without solving the VM. Real malware authors know this and either (a) encode the comparison values too, or (b) compute them at runtime from external sources.

Real-world tie-in: bytecode-VM obfuscation is alive and well in commercial protectors (VMProtect, Themida, Code Virtualizer). This is the toy version. The full version: 100+ opcodes, polymorphic dispatch (different VMs per build), and the verification target is computed by yet another VM. Real samples in the wild use this routinely.
